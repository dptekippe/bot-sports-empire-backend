**Daniel clarified:** "I know something is wrong with deepseek in the tui because when I change to deepseek I get a run error when I try to speak to you. But if you sent a request to deepseek you are fine"

**Key insight:** The TUI has its own DeepSeek configuration separate from the gateway.

**Found:** `~/.openclaw/models.json` has `deepseek-chat: {}` — an **empty object, no API key**.

**When Daniel switches to DeepSeek in TUI:**
1. TUI tries to use `deepseek-chat` model directly
2. Empty config = no API key sent
3. Result: HTTP 401

**When Roger (OpenClaw agent) uses DeepSeek:**
1. Uses key from `openclaw.json` (ends in `35d2`)
2. Works fine

**Root cause:** The TUI's `models.json` has an empty DeepSeek entry. The TUI doesn't pick up the API key from `openclaw.json`.

**Fix:** Update DeepSeek config via:
```bash
openclaw configure
```
Or:
```bash
openclaw models update deepseek --api-key "your-deepseek-key"
```

**Note:** The `openclaw-tui not found` error suggests Daniel tried to run `openclaw-tui` as a command, but it's not a standalone binary - it's the gateway client that connects to the OpenClaw gateway.
